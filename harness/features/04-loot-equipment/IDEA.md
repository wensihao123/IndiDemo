# IDEA — 暗黑式掉落与装备 (Loot & Equipment)

> 承 02:`Combat.loot_dropped(kind, rarity)` 事件已在产(kind ∈ 金/材料/装备,rarity ∈ 白/蓝/金),
> 但目前装备事件只飘个日志就没了。04 把"装备"这条变成真东西:有基底+词缀、进背包、自动填空槽。
> 承 03:战斗公式已扩出 **6 个维度**(攻击 / 攻速 / 护甲减伤 / 闪避 / 暴击率+倍率 / 每秒回血),
> 装备的基底+词缀直接喂这 6 维——原 IDEA 里"为了不动 02 而把基底压成纯 attack/生命"的妥协**已作废**。
> 这是支柱 2"刷→变强→够到更硬的怪"闭环里**"变强"那一环的物品载体**。
> **本次 jam(2026-06-18)在 03 收口后重刷**:大半旧 §8 线头已被 03 关掉或本轮敲定,详见 §8。
> 粗胚而已——刻意留洞,硬决策见 §8 交 Game Designer。

## 1. 一句话 / One-liner
小队挂机打怪不停掉装备,系统自动分解掉低稀有度、自动把对应部位的装备补进空槽,好货进背包;
玩家在悬浮窗里开一个**只读掉落包**翻看掉了什么(看不换),真正的换装留到 05 回城——
让"刷出来的战利品"实打实地转成"打得过下一关"的战斗力。

## 2. 玩家体验 / Fantasy
"我挂着它刷,空着的护甲槽刚才自动补上了一件蓝的、战士肉眼可见结实了点;偶尔点开掉落包翻翻——
'哦,掉了把金武器、三条词缀还带暴击!'——记下来,等回城再细挑换上。" 不被洪水淹(白装自动分解),
不被装备管理拖住(挂机期间零操作也在慢慢变强);参与感集中在**回城时少数几件好货的取舍**上。
是占便宜的小确幸 + 看着小队自己慢慢武装起来的陪伴感,负担留在城镇、挂机时纯看戏。

## 3. 核心循环(初稿) / Core loop (draft)
1. **挂机产出(自动,承 02):** 战斗击杀 → `loot_dropped` 掉装备 → 04 据 kind/rarity 生成"基底+词缀"实例。
2. **进包前过滤(自动):** 低于门槛的稀有度(默认白)→ **自动分解成材料**(喂 05 经济);够好的(蓝/金)→ 进背包。
3. **自动填空槽(自动,04 唯一的战力增益来源):** 某槽位为空且掉到/包里有该部位装备 → **自动穿上**;
   **绝不自动替换已穿戴的**。裸装战士随挂机从空槽逐步填满,微变强。
4. **只读查阅(手动,04 的玩家介入点 = 仅"看"):** 悬浮窗点开**掉落包面板** → 浏览掉了哪些装备、各自基底+词缀。
   **只能看,不能换**(方便直接验证掉落/生成在工作)。
5. **变强 → 回到循环:** 自动填空 + (05 的)回城换装让战士更猛 → 啃得动更硬的关 → 更深的关掉更好的装备。
   → 重复理由:下一关的门槛(02 的 Boss/场景)就是"够不着的怪",更好的装备是够到它的钥匙。

> **边界(本轮敲定):** 04 = 掉落 + 物品数据 + 自动分解 + 自动填空槽 + 只读掉落包。
> **手动换装 + 对比面板 + 打造/强化消耗材料 = 05 城镇。** 详见 §8 边界 flag。

## 4. 关键规则与状态 / Key rules & states
**装备 = 基底 + 词缀(本轮敲定模型):**
- **基底专属轴 + 词缀通吃。** 基底随**部位**固定为该部位的"招牌维度",给身份;词缀池可 roll **任意 6 维**,给惊喜。
- **三个槽位 + 招牌基底:**
  - **武器** 基底 = 攻击 + 攻速
  - **护甲** 基底 = 护甲(减伤)
  - **饰品** 基底 = **生命 / 闪避 / 每秒回血,三选一**(每件掉落时随机定一个招牌轴)
- **词缀池 = 6 维通吃**:攻击 / 攻速 / 护甲 / 闪避 / 暴击(率或倍率)/ 每秒回血,任意部位都可能 roll 到任意词缀
  (经典暗黑式:武器上也可能 roll 出血量/闪避词缀)。具体可 roll 哪些、各维数值区间待 Game Designer 定(§8)。
- **稀有度决定词缀条数:** 白 = 仅基底(0 词缀)/ 蓝 = 1-2 词缀 / 金 = 3+ 词缀。数值高低细则待 Game Designer 定。

**换装/穿戴规则:**
- **04 内只有"自动填空槽"**:空槽位 → 对应部位掉落/包内物品自动补上;**绝不自动替换已穿戴的装备。**
  (理由:词缀/未来套装会让"哪件更好"无法机械判定,自动替换可能纸面变强实则变弱——见 §8。)
- **手动换装(玩家亲挑每槽穿哪件)+ 逐项差值对比面板 = 05 城镇做**,不在 04。

**背包与洪水:**
- 后台持续掉落 → **自动分解低稀有度**(玩家门槛,默认白装→材料)→ 只有够好的进包,洪水自消化,
  不逼玩家频繁清包(守支柱 1)。

**与战斗的接口(03 后已通):** 穿戴的基底+词缀汇总成战士的 6 维战斗属性,喂给 03 扩好的解算
(`PartyMember` 已有 attack / attack_speed / armor / dodge_chance / crit_chance / crit_mult / hp_regen 字段)。
原 IDEA 标的"02 容不下攻速/护甲"问题**已被 03 解决**,不再是阻塞。

## 5. 反馈 / juice 意图 / Feedback & juice intent
- **掉落瞬间(已有,02):** 蓝/金掉落出对应色光柱 + 金装窗口内一闪;白装默默(现在还会自动分解)。
- **自动填空槽时:** 给个轻提示(余光可见,如战士某槽位"叮"地亮一下 / 一行日志"自动装备:蓝护甲"),
  让玩家知道挂机期间小队在自己武装起来。不弹 OS 通知(守支柱 1)。
- **进包提醒:** 有够好的新装备进掉落包时给个轻提示(角标/微光),不打扰。
- **掉落包面板(只读):** 列出包内每件的基底+词缀(分稀有度色)。**04 不做"对比已穿戴"的绿↑红↓**——
  那是 05 换装面板的核心反馈;04 的面板只回答"我刷到了什么",不回答"该不该换"。
- **自动填空后:** 战士属性数字跳变;战斗里也能间接感到(怪死更快/血条更厚)——主反馈是属性+提示,不强求战斗演出。

## 6. 最小版本 / Minimal version
能跑通"掉→分解/自动填空/进包→只读查阅"一条线的最小切片(换装闭环的"看"的一半,"换"的一半归 05):
- 三槽位(武器/护甲/饰品)+ 装备 = 基底+词缀数据结构;白/蓝/金 → 0 / 1-2 / 3+ 词缀。
- 基底直接映射到 03 的 6 维(武器攻击+攻速、护甲护甲、饰品三选一);词缀池**先挑 2-3 维起步**(具体哪几维交 GD),
  数据结构按"可 roll 任意 6 维"设计,避免日后扩池重写。
- 自动分解白装→材料(门槛先写死默认值,可配置 UI 留后);自动填空槽;只读掉落包面板(列基底+词缀,不可换)。
- 背包/掉落包一个朴素列表即可;穿戴属性汇总进 03 战士。
- **先不做(归 05 或更后):** 手动换装、对比面板、打造/强化消耗材料、套装、宝石、词缀重铸、
  按词缀的自动比较/自动替换、可调分解门槛 UI。

## 7. 与游戏支柱的关系 / Pillar fit
- **支柱 2(变强够到够不着的怪,权重 3)——主服务对象。** 04 是"变强"的物品载体;自动填空槽让挂机本身就推进变强,
  回城手动换装(05)是支柱 2 的"主动构筑"高光时刻。**张力点:** 04 把"主动取舍"挪到了 05,04 自身的支柱 2 体验偏被动——
  这是有意的(04 先把掉落/物品/数据跑通),但要确认 04 单独 playtest 时"够不够有意思"还是必须连着 05 才成立(见 §8)。
- **支柱 1(伙伴非任务,权重 5)——靠自动分解 + 自动填空槽 + 只读包守住。** 挂机期间零必做操作、洪水自消化、
  战士自己慢慢武装;掉落包"想看才看"。**张力点:** 自动填空槽的提示强度、进包提醒频率要克制,别变成频繁打扰。
- **支柱 3(惊喜非赌场,权重 2)——靠稀有度=词缀数 + 词缀通吃承载。** "金的!三条词缀还带暴击!"的小确幸;
  白装自动分解后,留在包里的本就稀少,每件都值得点开看一眼。不靠强随机逼氪。

**仍待 Game Designer 拍板:**
1. **词缀 roll 细则。** 词缀池虽定"通吃 6 维",但具体:每个部位能 roll 哪些维(全开?还是部位有偏好池)?
   各维数值区间?暴击词缀 roll 的是率还是倍率(还是两种各算一条)?攻速作为词缀时步长?
2. **稀有度数值梯度。** 白0/蓝1-2/金3+ 的词缀条数已定,但每条词缀的数值范围、是否随稀有度抬高、基底数值是否也随稀有度浮动,待定。
3. **战士开局空装还是自带基础装?** 影响"自动填空槽"早期出场频率与新手手感——现在 04 唯一的变强来源就是自动填空,
   开局全空会让前几场"叮叮叮自动装备"很热闹,自带基础装则前期更平。用户问卷未答,留 GD 定。
4. **04 单独 playtest 的乐趣验证。** 换装挪到 05 后,04 自身的玩家介入只剩"看掉落包"。需确认 04 切片单独跑时
   是否"够看出循环成立",还是必须等 05 一起才算验证了支柱 2(见 §7 张力点)。可能影响 04 是否需要一点点临时换装入口来自测。
5. **分解门槛的默认值与可配置度。** v1 默认"白装自动分解"够不够?要不要让玩家调(蓝也分解/只留金)?默认值要不淹包又不误删好货。
6. **分解产出什么材料、喂给谁。** 04 产材料 vs 05 消费材料的接口边界(类比 02→04 的 `loot_dropped` 边界)——
   建议 04 只产"得到 X 材料"事件/数据 + 存起来,打造消耗归 05。材料种类/数量公式交 GD。
7. **自动填空槽 + 未来套装/词缀的兼容(标准注记)。** v1 已定不自动替换已穿戴的(怕纸面变强实则变弱/破坏套装)。
   即便日后做"智能自动换",也得先有一套能表达"实际强度/套装意图"的评估,别退化成纯纸面比大小。
8. **掉落包/背包满了的兜底。** 自动分解后理论上不爆,但万一玩家(05 做了可调门槛后)调到"全留",仍需兜底(停收/最旧自动分解?)。
